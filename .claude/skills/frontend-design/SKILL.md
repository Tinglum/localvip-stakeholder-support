# Frontend Design Skill

Create frontend work that feels deliberately designed, premium, memorable, and production-grade.

The goal is not merely to make UI functional.
The goal is to make it feel **designed at Apple-level quality**, with distinctive visual character, refined interaction, and strong product taste.

---

## Core Principle

Every frontend task should improve the perceived quality of the product.

Even small UI requests should be treated as design opportunities.

If the user asks for:

* a button
* a modal
* a card
* a form
* a dashboard panel
* a navigation item
* a page section

Do not produce the bare minimum.

Create something refined, elevated, and visually memorable, while still fitting the existing product.

---

## Design Ambition

Default to **premium, standout design**, not safe generic UI.

The output should feel:

* polished
* dimensional
* intentional
* elegant
* modern
* distinctive
* tactile
* production-ready

Avoid:

* plain default buttons
* flat boring cards
* generic SaaS layouts
* template-looking dashboards
* random gradients
* AI-looking UI
* visual decisions without purpose

---

## Codebase Respect

Before changing UI:

1. Inspect existing components and styling
2. Understand the current visual system
3. Preserve functionality
4. Improve the design without breaking the product
5. Extend existing patterns where strong
6. Upgrade existing patterns where weak

Do not rewrite unrelated code.

---

## Design Thinking Before Coding

For each frontend task, quickly determine:

* **Purpose**: What should this element help the user do?
* **Context**: Where does it live in the product?
* **Hierarchy**: How important is it visually?
* **Interaction**: What should happen on hover, focus, click, open, close?
* **Memorability**: What detail makes it feel custom?
* **Restraint**: What should stay simple so the design does not become noisy?

---

## High-Design Requirements

When appropriate, add:

* layered depth
* shadow systems
* glass, blur, or soft material effects
* gradient mesh or glow effects
* subtle grain or texture
* refined hover states
* animated reveal states
* micro-interactions
* strong typography hierarchy
* asymmetric composition
* custom empty states
* elegant loading states
* polished focus states
* tactile transitions

These should complement the product, not feel pasted on.

---

## Interaction Standard

Interactive elements should feel alive.

Buttons, cards, modals, menus, and panels should include:

* hover state
* active state
* focus-visible state
* disabled state where relevant
* transition timing
* clear affordance
* accessible contrast

For important actions, consider:

* elevated CTA treatment
* animated border or glow
* layered background
* icon motion
* subtle press effect
* contextual tooltip or supporting text

---

## Modal / Overlay Standard

If designing a modal or overlay, avoid basic centered boxes.

Prefer:

* layered backdrop
* blur or depth
* entry animation
* refined shadow
* visible hierarchy
* clear escape/close pattern
* responsive sizing
* accessible focus handling
* strong primary action

A modal should feel like an intentional product moment, not a default dialog.

---

## Button Standard

If designing a button, avoid plain rectangles.

A high-quality button may include:

* layered surface
* tasteful gradient or material effect
* border highlight
* hover lift or glow
* icon movement
* press compression
* strong text hierarchy
* accessible focus ring

The button must still be usable, readable, and consistent with the product.

---

## Layout Standard

Avoid predictable, uniform layouts.

Prefer:

* hierarchy-driven composition
* asymmetric balance
* editorial spacing
* layered panels
* intentional negative space
* controlled density
* visually distinct sections

Do not make every card the same size unless the content truly requires it.

---

## Typography Standard

Typography should carry design quality.

Use:

* strong heading hierarchy
* refined body rhythm
* meaningful contrast
* carefully chosen font pairings when possible

Avoid:

* generic typography-only layouts
* system-font sameness unless the existing project requires it

---

## Color Standard

Use color with confidence and restraint.

Prefer:

* controlled palettes
* dominant base color
* 1–2 accent colors
* semantic status colors
* CSS variables or design tokens where possible

Avoid:

* random colors
* loud gradients without purpose
* purple SaaS clichés
* low-contrast decoration

---

## Motion Standard

Use motion to make the product feel refined.

Good motion:

* reveals hierarchy
* gives feedback
* makes transitions feel premium
* supports usability

Avoid:

* constant motion
* distracting animation
* gimmicks that slow the user down

Use CSS transitions by default. Use animation libraries only if already present or clearly justified.

---

## Scope Control

High-design does not mean uncontrolled scope.

Do:

* improve the requested area deeply
* make related small improvements if needed
* preserve the rest of the system
* keep implementation maintainable

Do not:

* redesign unrelated pages
* replace architecture unnecessarily
* generate massive files without reason
* overbuild hidden complexity

---

## Implementation Rules

* Build real working code
* Preserve existing behavior
* Use existing framework conventions
* Keep components modular
* Make responsive layouts
* Include accessible states
* Use realistic mock data only when needed
* Avoid pseudo-code unless requested

---

## Cost Control

Be efficient, but not bland.

* Do not repeat long explanations
* Do not regenerate entire files unless needed
* Prefer surgical edits with premium execution
* Output only changed files or sections unless full files are requested
* Spend tokens on design quality and implementation, not filler text

---

## Output Contract

When implementing frontend work, return:

1. Brief design direction
2. Files changed
3. Key implementation notes
4. Checks run, if any
5. Short summary

Do not write long essays unless requested.

---

## Final Standard

Every frontend change should make the product feel more valuable.

If the result looks like a normal SaaS template, it failed.

If a small component feels like it belongs in a premium product, it succeeded.
